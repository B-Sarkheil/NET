# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Namvaran Excel Tracker (NET) — a small Flask app that scans a network folder structure and renders it as an interactive flowchart in the browser. There is no database; all state comes from folder/file layout on disk plus `Setting.xlsx` workbooks read with `openpyxl`.

## Commands

No test suite or linter exists in this repo — don't invent one.

Dependencies are declared in `src/requirements.txt` (`flask`, `openpyxl`): `pip install -r src/requirements.txt`

Run the dev server:
```
cd src
python app.py
```
(or double-click `src/start.bat`, which also prints the LAN IP). Serves on `http://localhost:5000` with `debug=True` but `use_reloader=False` — **editing `app.py` requires manually stopping and restarting the process**; it will not pick up backend changes on its own. Template (`templates/index.html`) changes *do* auto-reload per request. Hitting `GET /reload` re-scans folders/Setting.xlsx files into the cache without a restart, but still won't reload Python code changes.

Build the standalone Windows exe:
```
deploy\build.bat
```
Installs `src/requirements.txt` + PyInstaller, then runs PyInstaller (`--onedir`) against `src\app.py` (CLI flags, not the checked-in `deploy\NET.spec` directly — PyInstaller regenerates that spec file from these flags every run, so **edit `build.bat`'s flags, not `NET.spec`, for lasting changes**; `NET.spec` is kept in sync by hand for reference/direct-spec builds only). Bundles `src\templates` and `src\static`, explicitly collects `openpyxl`'s submodules (`--collect-submodules openpyxl` — PyInstaller's static analysis can miss its lazily-imported reader/writer modules), and outputs to `product\NET\` (then copies `src\config.txt` alongside the exe). To distribute: copy the whole `product\NET\` folder, edit its `config.txt`, run `NET.exe`.

## Layout

- `src/` — the actual app: `app.py` (all backend logic, single file), `templates/index.html` (all HTML+CSS+JS, single file, no separate static JS/CSS beyond `static/favicon.ico`), `config.txt`, `start.bat`
- `product/NET/` — build output (gitignore-worthy; regenerated by `deploy/build.bat`)
- `deploy/` — `build.bat` + `NET.spec` (PyInstaller spec)
- `simulation/` — sample data used as dev `SOURCE_PATH`/`COMMIT_PATH` (`fs/` and `fs - files/`)

`config.txt` lives next to `app.py` (or next to `NET.exe` when frozen) and is read at startup via `load_config()`; missing keys fall back to hardcoded defaults pointing at `D:\Behnam\NET\...` (stale paths from before the repo was split into `src/product/deploy` — don't rely on the fallback, always keep `config.txt` current).

## Core architecture: two roots, three settings tiers

Everything hinges on two configured paths that serve **different purposes**, plus a fixed convention for the `Setting.xlsx` workbooks inside them:

- **`COMMIT_PATH`** is the single source of truth for *which folders/subfolders exist*. `get_structure()` (builds the on-screen boxes) and `get_all_subfolder_keys()` (used by connection-resolution and `/fetch-data`) both enumerate `COMMIT_PATH`. This exists because some folders (e.g. a `Pump DS` folder) may never be set up under `SOURCE_PATH` at all but still need to appear and participate in connections.
- **`SOURCE_PATH`** is where the *primary* `Setting.xlsx` for each folder normally lives.

`load_all_settings()` (run once at startup into `SETTINGS_CACHE`, keyed `"Folder/Sub"`) resolves each `COMMIT_PATH` key's settings in priority order:
1. **Primary**: `SOURCE_PATH/Folder/Sub/Source/Setting.xlsx` → feeds Access (In Charge), Notification, and Fetch (connections). Marks `has_setting: True`.
2. **Fallback**: `COMMIT_PATH/Folder/Sub/Raw Files/Setting.xlsx` (only checked when primary is missing) → feeds **Fetch/connections only**; Access/Notification are forced empty and `has_setting: False`, so the frontend hides those two modal sections for folders with no real `SOURCE_PATH` setup. This is also why a folder literally named `Raw Files` is filtered out of the Commits file listing in `/files` — it's settings metadata, not a committed deliverable.
3. **Nested** (unrelated third case): folders a user drills into inside the Commits file browser (e.g. `Pump DS/P-2001`) can carry their own `Source/Setting.xlsx`, read on demand (not cached) via `_find_nested_setting()` — feeds only that nested folder's own In Charge/Notification, never connections. `/access`, `/notification`, `/files`, and `/download` all accept an optional `path` query param for this, sanitized against traversal by `_safe_subpath()`.

`parse_setting()` is the one place that reads a workbook: `access` from the `Access` sheet (col B), `ref_paths`/`fetch_headers`/`fetch_rows_by_ref` from `Fetch` (path in col J), `generate_paths`/`generate_headers`/`generate_rows_by_ref` from `Generator` (path in col G — note the sheet is named "Generator", not "Generate"), `notification_headers`/`notification_rows` from `Notification`. `build_connections()` turns both into `{from, to, type}` arrows by resolving each path against `get_all_subfolder_keys()` — but in **opposite directions**: a `Fetch` path is the `from` (this folder pulls from it, so it's the connection's source; the file's own owner is the `to`; `type: 'Fetch'`), while a `Generator` path is the `to` (this folder produces output that goes there; the file's own owner is the `from`; `type: 'Generate'`).

Clicking a connection arrow opens a popup showing the underlying spreadsheet rows: `/fetch-data` for `type: 'Fetch'` (owner = `to_parent`/`to_sub`), `/generate-data` for `type: 'Generate'` (owner = `from_parent`/`from_sub`) — `openConnection(fromKey, toKey, type)` in `index.html` picks the endpoint.

## Frontend (single template, no build step)

`templates/index.html` contains the CSS grid layout (top / left+Process+right / bottom, colored boxes) plus a modal opened via double-click on a box. The modal has three independently-hidden sections (In Charge / Commits / Notification) driven by `has_setting`. The Commits section supports drilling into nested folders (`loadCommits()`, `enterCommitFolder()`, `goUpCommits()` in the inline `<script>`), and re-fetches In Charge/Notification for whatever nested level is currently open (`refreshSettingsPanels()`) — so the same modal shows different settings data depending on how deep you've navigated.

## Known operational gotcha

Because `use_reloader=False`, a `python app.py` process can be left running (e.g. backgrounded, or orphaned from a killed terminal) and keep holding port 5000 with stale code while a newer process fails to bind or coexists via Windows' `SO_REUSEADDR`. If behavior doesn't match the current source, check `netstat -ano | findstr :5000` for more than one PID before debugging the code itself.
